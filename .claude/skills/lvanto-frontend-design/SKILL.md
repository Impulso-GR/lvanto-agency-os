# Lvanto Frontend Design Skill

## 1. Purpose

Guide visual and interface decisions for Lvanto client projects. Translate approved copy and strategy into clear layout direction, design review, and implementation-ready guidance for landing pages and websites.

---

## 2. When to Use

- Before starting frontend implementation on a landing page or website.
- When reviewing a design or layout for visual quality and conversion alignment.
- During UI improvement or redesign work.
- Before delivering a project to the client (pre-delivery QA).
- When a design feels generic, cluttered, or commercially weak.

---

## 3. Inputs to Request

Before reviewing or directing, ask for:

- **Project type**: landing page, website, section, or component?
- **Client brand**: colors, fonts, tone, existing references?
- **Target audience**: who will see this and what should they feel?
- **Conversion goal**: what action should the page drive?
- **Current design or wireframe**: screenshot, URL, or description?
- **Implementation constraints**: HTML/CSS/JS only? Framework? CMS?
- **Language**: Spanish (default) or other?

---

## 4. Frontend Design Review Framework

Apply this checklist when reviewing or directing any visual/interface output:

1. **Visual hierarchy** — Is it immediately clear what the most important element is? Does the eye flow naturally from headline to CTA?
2. **Section structure** — Are sections logically ordered? Does each section serve one clear purpose?
3. **Spacing and rhythm** — Is there enough whitespace? Do sections breathe or feel cluttered?
4. **CTA visibility** — Is the primary CTA prominent, above the fold, and visually distinct?
5. **Responsive layout** — Does the layout work on mobile without horizontal scroll, text overlap, or broken sections?
6. **Brand consistency** — Are colors, fonts, and tone consistent across all sections?
7. **Accessibility basics** — Sufficient color contrast, readable font sizes (min 16px body), and descriptive link/button text.
8. **Premium perception** — Does the page look professional and credible? Would a potential client trust it?
9. **Implementation feasibility** — Can this be built cleanly in HTML/CSS/JS without excessive complexity?

---

## 5. Output Formats

### Landing page visual direction
- Recommended layout structure (section order and purpose)
- Hierarchy notes per section (headline size, CTA placement, image use)
- Color and font application guidance

### Website section review
- Section-by-section notes: what works, what to fix, and why
- Prioritized: critical fixes first, polish second

### UI improvement checklist
- Bullet list of specific, actionable improvements
- Label each as: Critical / Important / Optional

### Responsive review
- Mobile-first observations
- Breakpoint recommendations (mobile, tablet, desktop)
- Specific layout adjustments needed

### Pre-delivery design QA
- Pass/Fail per framework item (hierarchy, spacing, CTA, responsive, brand, accessibility, premium feel)
- Summary: ready to deliver, or list of required fixes

### Frontend implementation guidance
- HTML structure recommendations (semantic elements, section order)
- CSS notes (spacing scale, font pairing, color variables)
- Interaction notes (hover states, transitions) — only if requested

---

## 6. Quality Rules

- **Clarity over decoration**: remove anything that does not serve the conversion goal or the user's understanding.
- **Support conversion goals**: every design decision should make the primary CTA easier to reach.
- **Maintain brand coherence**: do not mix styles, introduce new colors, or use fonts outside the client's system.
- **Avoid generic templates**: flag and fix any layout that looks like a default Bootstrap or theme pattern.
- **Feasibility first**: only recommend what can be implemented cleanly in HTML/CSS/JS or the specified stack.
- **Write in Spanish by default** unless the operator requests another language.
